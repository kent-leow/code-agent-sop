# Requirements-to-Code Workflow

An agent-driven pipeline for transforming raw business requirements into production-ready code with full traceability.

## Overview

```
Raw Requirements
  → @generate-plan          →  plan.md  +  Jira story
  → @refine-plan            →  resolved open questions, locked plan
  → @generate-execute-plan  →  execute-plan-001.md, 002.md, ...
  → @execute-plan           →  production code + tests
```

Each stage produces a concrete artefact in `.docs/<task-name>/`. Nothing moves forward until the previous output is complete and verified.

## File Structure

```
.docs/
└── <task-name>/              # One folder per feature / task
    ├── plan.md               # Stage 1 output: requirements, AC, open questions
    ├── jira.json             # Jira story + sub-task keys
    ├── execute-plan-001.md   # Stage 2 output: vertical slice 1
    ├── execute-plan-002.md   # Stage 2 output: vertical slice 2
    └── figma/                # Cached Figma assets (auto-created by agents)
        ├── <nodeId>.json
        ├── <nodeId>.png
        └── <nodeId>.md
```

## Stages

### Stage 1 — Plan
**Agent**: `@generate-plan`
- **Input**: Raw requirement, feature request, or user description
- **Output**: `.docs/<task-name>/plan.md` + Jira story (via `jira-ticket` skill)
- **plan.md sections**: Summary, Scope, Acceptance Criteria, Open Questions, Changelog

### Stage 1b — Refine (if open questions exist)
**Agent**: `@refine-plan`
- **Input**: `.docs/<task-name>/plan.md` + answers to open questions
- **Output**: Updated `plan.md` with resolved questions, cascaded into any execute plans
- Readiness check is performed — plan must be clear before proceeding

### Stage 2 — Generate Execute Plans
**Agent**: `@generate-execute-plan`
- **Input**: `.docs/<task-name>/plan.md` (must be ready — no unresolved blocking questions)
- **Output**: `execute-plan-001.md`, `execute-plan-002.md`, … (one per vertical slice)
- Each file contains: task list, file-level checklist, code guidance, test requirements, Done When criteria

### Stage 2b — Refine Execute Plan (if corrections needed)
**Agent**: `@refine-execute-plan`
- **Input**: `execute-plan-NNN.md` + corrections / additions
- **Output**: Updated execute plan with changelog entry

### Stage 3 — Execute
**Agent**: `@execute-plan`
- **Input**: `execute-plan-NNN.md`
- **Output**: Production code, tests, all checkboxes marked `[x]`
- Checks prerequisites (earlier slices must be Done) before starting
- Syncs sibling execute plans for cross-slice impact
- Creates Jira sub-task and logs it in `jira.json`

## Naming Conventions

| Artefact | Pattern | Example |
|---|---|---|

| Execute plan | `execute-plan-NNN.md` | `execute-plan-001.md` |
| Jira state | `jira.json` | `.docs/<task>/jira.json` |

## Integrations

### Jira
- `@generate-plan` auto-creates a Jira story after writing `plan.md`
- `@execute-plan` auto-creates a Jira sub-task per slice
- All keys stored in `jira.json` for traceability
- Requires: `JIRA_TOKEN`, `JIRA_EMAIL`, `JIRA_BASE_URL`, `JIRA_PROJECT_KEY` in `~/.zshenv`

### Figma
- Any agent that encounters a Figma URL fetches and caches design context under `figma/`
- Cache-first: existing `<nodeId>.json` is reused unless a refresh is signalled
- Requires: `FIGMA_TOKEN` in `~/.zshenv` (fallback when Figma MCP is unavailable)

## Quality Gates

- **Traceability**: every execute-plan task links back to an AC row in `plan.md`
- **Prerequisites**: `@execute-plan` refuses to start if earlier slices have open Done When items
- **Test coverage**: every logic task in an execute plan must have a test child task
- **Changelog**: every refinement appends a dated entry to the affected file
